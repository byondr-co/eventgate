### Plan B — Accounts, Orgs, Memberships, Magic-link

Custom email-only `User` model (no usernames, no passwords by default), `MagicLinkToken` with SHA-256-at-rest + 15-min TTL + single-use, JWT in httpOnly cookies (`CookieJWTAuthentication`), `Organization` + `OrganizationMembership` + `Invite` models, `OrgScopedModel` abstract base + `IsOrgMember` / `HasOrgRole` permissions, members + invite endpoints. Frontend: `/login`, `/auth/callback`, `/`, `/orgs/new`, `/orgs/[slug]`, `/orgs/[slug]/members`, `/invites/[token]`.

### Plan C — Events, Public Registration, QR + Email

`apps.events` (`Event` + `RegistrationField` with EN+KM labels) and `apps.guests` (`Guest` inheriting `OrgScopedModel`, separate `entry_status` / `info_status` per brief Appendix A). Public anonymous registration at `POST /api/v1/e/<org>/<event>/register/` returns guest_id only (token never echoed). QR rendered on demand via `segno` at `GET /api/v1/guests/<id>/qr.png?token=<raw>` with constant-time compare. Magic-link AND QR delivery routed through Celery tasks (run eagerly in web process — see below). Resend wired conditionally. `NotificationDispatch` model logs every send. Frontend: public `/e/[orgSlug]/[eventSlug]/register` + `/registered/[guestId]`, organizer event create wizard + events list + event detail + form builder + guest list. `next-intl` with EN + KM message bundles.

### Plan D — Walk-in Flow + Pre-registered Scanner PWA + Celery Worker + Public Event Detail

New backend apps: `apps.audit` (skeleton — append-only `AuditEvent`, `write_audit()` is the only sanctioned writer; UI viewer lands in Plan F), `apps.devices` (`ScannerDevice` with SHA-256-at-rest device tokens + role gating, `EventPinSession` 8h TTL, `DeviceTokenAuthentication` + `SessionTokenAuthentication` reading the `Authorization` header), `apps.checkins` (one hot-path endpoint `POST /api/v1/checkins/` with `pg_advisory_xact_lock` + Redis-backed idempotency + audit), `apps.walkins` (display-next + public claim + public info; partial unique index `one_displayed_walkin_per_scope` is the backstop). New modules: `apps.guests.transitions` (single source of truth for `entry_status` / `info_status` moves), `apps.common.locks`, `apps.common.idempotency`. Existing `apps.events` gained PIN set/rotate (bcrypt-hashed, owner/admin only) + `PublicEventDetailView`. Frontend `(scanner)` route group lives at `/scanner/{enroll,unlock,scan,walkin}` with PWA manifest + minimal static-asset service worker (Workbox + IndexedDB defers to Plan E). Public `/e/<org>/<event>/{claim,info}/<token>/` pages. Camera uses `BarcodeDetector` with a manual-entry fallback for browsers that lack it. Walk-in QR rendered via `qrcode.react` SVG at 2048px scaled via CSS to 85% of the viewport. Celery worker process group on Fly (`[processes].worker`) running its own Machine; `CELERY_TASK_ALWAYS_EAGER` Fly secret unset; `PUBLIC_BASE_URL` Fly secret set so walk-in QRs encode the right host.

### Plan E — Offline Scanner Sync (Workbox + IndexedDB + Mutation Queue + Conflict Routing)

Replaces Plan D's minimal static-only SW with a Workbox-composed service worker bundled via esbuild (`sw-src/sw.ts` → `public/sw.js` via `scripts/build-sw.mjs`, wired into `pnpm build`). Adds a Dexie 4 IndexedDB layer at `lib/scanner/db.ts` with three stores: `guests` (minimal projection — `id`, `entry_token`, `full_name`, `email`, `guest_type`, `entry_status`, `info_status`, `updated_at`), `mutation_queue` (offline check-in writes with status enum `pending | in_flight | completed | conflict | failed | escalated`, exponential backoff 1s→60s × 8 max attempts), `meta` (key/value for sync cursor + ETag).

The cache primes on PIN unlock from a new backend endpoint `GET /api/v1/orgs/<org>/events/<event>/guests/sync/?since=<iso>` (scanner-session auth, ETag-aware, returns 304 when nothing changed). It refreshes on `online` / `visibilitychange` / a 5-minute interval. The scan page branches on `navigator.onLine`: online uses the existing `POST /api/v1/checkins/`; offline looks up the token in IndexedDB, renders an optimistic outcome (success / duplicate / "will validate on reconnect"), and enqueues the mutation. A separate sync loop (`online` + `visibilitychange` + 30s heartbeat) drains the queue.

Server-authoritative conflict resolution: when a queued mutation reconnects and the server returns 409 from a *different* device/gate, the client marks it `conflict` and the server emits an additional `checkin.conflict` audit row alongside the existing `checkin.duplicate`. A new `POST /api/v1/scanner/escalations/` endpoint (in new `apps.scanner` Django app) writes a `checkin.help_desk_escalation` audit row when the staffer taps "Send to help desk" on the new `/scanner/escalations` page. Plan F builds the inbox that reads those rows.

Scanner header gains: an "Install" PWA prompt button (capturing `beforeinstallprompt`), a "⚠ N conflict(s)" link to `/scanner/escalations` when conflicts exist, and "● offline — N queued" pill. Persistent below-header banner appears when offline. Browser Sentry SDK init dynamically loads from `sentry.client.config.ts` only on `/scanner/*` routes (first frontend Sentry land — needs `NEXT_PUBLIC_SENTRY_DSN` set on Vercel before it activates).

**Plan E ops cleanups (Task 0 wave):** `worker.restart=always` declared explicitly in `fly.toml`; **Vercel `rootDirectory` set to `frontend` via REST API** (root cause of "auto-deploy not working" during Plan D: builds were firing but failing with `NEXT_NO_VERSION` because `rootDirectory` was null — finding documented at `PLAN_E_TASK_0B_FINDINGS.md`); `prettier` pinned exact (no caret) in `frontend/package.json`; all 7 Plan-D-era `# type: ignore` comments replaced with real annotations (parameterized `BaseUserManager["User"]`, `models.Manager["Organization"]`, `TYPE_CHECKING` forward refs, two `cast(timedelta, settings.SIMPLE_JWT[...])`).

## Working agreements (preserved across sessions)

### Brief Appendix A — firm product decisions (do NOT change without explicit reason)

- Pre-registered and walk-in flows stay separate.
- Pre-registered QR encodes the **raw `entry_token` only** (no URL wrapping).
- Pre-registered check-in mutation requires an **authorized staff scanner session**.
- Walk-in QR is **unique per guest** and persisted before display.
- Walk-in guests enter first, then complete info inside the hall.
- `entry_status` and `info_status` are **separate fields** with explicit transitions.
- No automatic walk-in token expiry.
- Duplicate / invalid / unauthorized cases route to **help desk lane**.
- Public root URL is the registration flow only; staff routes are gated.

### Engineering method

- **Brainstorm → write a plan → execute task-by-task.** Plans live in `docs/plans/YYYY-MM-DD-plan-X-name.md`. Each task has `- [ ]` checkboxes with explicit step-by-step code, file paths, commands, and expected output.
- **TDD discipline for backend.** Red → green → commit. Every task has a `tests/test_*.py` file. No skipping the red step.
- **Hybrid execution for plans of 20+ tasks.** Backend (TDD-shaped) tasks dispatched via Agent tool (`general-purpose`, sonnet) with full task text inline (never make the subagent read the plan). Frontend (component build) and deploy tasks done inline by the controller. After each backend task: spec-compliance review (sonnet) → code-quality review (sonnet) → fix → mark complete.
- **Frequent commits.** One commit per task, conventional-commit prefix (`feat(events): …`, `fix(cors): …`, `test(orgs): …`, `docs(plan-d): …`).
- **Pre-commit hooks** run ruff + ruff-format + ESLint + trailing-whitespace etc. If a hook modifies files, re-stage and re-commit; never use `--no-verify`.
- **Plans are living docs.** When execution surfaces a deviation, the implementer adjusts the code AND notes the deviation in the plan's completion log at the end. We don't silently diverge.
- **No new docs unless explicitly requested.** Stick to plans + commit messages.
